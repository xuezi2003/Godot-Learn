# C5-P0：章节概述

> 本集介绍第五章”敌人系统”的整体设计思路，比较了几种常见的敌人 AI 方案，并确定采用“有限状态机 + 决策层 + 传感器”的组合式架构。


## 1. 本章目标

- 完全专注于创建敌人与敌人 AI。
- 从最简单的敌人开始，逐步构建可复用的模块化系统。
- 关键问题：如何为敌人做决策？


## 2. 敌人 AI 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **基础敌人脚本**（一个脚本内硬编码） | 简单、可扩展、适合小型游戏或 Game Jam | 敌人数量多或差异大时难以维护 |
| **有限状态机（FSM）** | 结构清晰，状态可复用、可混合搭配 | 状态数量可能激增，决策能力弱 |
| **行为树（Behavior Tree）** | 模块化决策，可添加优先级，每帧决策 | 偏抽象，实现复杂，有点“过度设计” |
| **目标导向行动规划（GOAP）** | 能涌现不可预测的玩法 | 实现庞大复杂；类银河恶魔城游戏更希望敌人行为可预测 |


## 3. 最终采用的折中方案

1. **基础敌人脚本**：提供生命值、移动速度、死亡音效等通用属性。
2. **有限状态机**：管理行走、眩晕、死亡、攻击、追逐等状态，状态本身不做复杂决策。
3. **决策层（Decision Engine）**：根据传感器信息决定切换到哪个状态；每种敌人可自定义覆盖。
4. **黑板（Blackboard）**：集中存储敌人数据（生命值、目标、距离、是否发现边缘等）。
5. **传感器（Sensors）**：视野锥、边缘检测、听觉等，即插即用。
6. **组合式设计**：通过节点树组合功能，而不是大量继承。


## 4. 关键节点/组件

- `DamageArea` / `HazardArea`：伤害区域与危险区域。
- `EdgeSensor`：边缘检测器，防止敌人掉下平台。
- `PlayerSensor`：玩家探测器/视野锥。
- 敌人状态机：`EnemyStateMachine`、`EnemyState`。
- `Blackboard`：数据资源。
- `DecisionEngine`：决策引擎。


## 5. 学习路线

1. 先做一个硬编码的史莱姆敌人，理解最基础的需求。
2. 再用新系统（状态机 + 决策层）重建史莱姆。
3. 逐步加入传感器、粒子、战利品等模块。
4. 最后用社区投稿的敌人（如 Living Armor、Demon Chest、Doom Scribe）做完整案例。
