# C2-P10：玩家生成与关卡构建指南

> 修复关卡过渡的默认尺寸问题，创建玩家出生点节点，并分享关卡原型设计与能力锁设计的建议。

## 1. 修复关卡过渡门默认尺寸问题

- 在 `LevelTransition` 中，`apply_area_settings()` 只在 `size` 或 `location` 改变时被调用。
- 如果保留默认值，运行时不会应用区域设置，导致过渡门碰撞范围错误。
- **修复**：在 `_ready()` 开头直接调用 `apply_area_settings()`。

```gdscript
func _ready():
    if Engine.is_editor_hint():
        return
    apply_area_settings()
    # 后续连接信号...
```

## 2. 玩家出生点（PlayerSpawn）

### 设计动机

- 将玩家从每个关卡场景中移除，避免重复维护多个玩家实例。
- 运行时如果关卡中还没有玩家，由 `PlayerSpawn` 动态创建；若已存在则不重复生成。

### 场景结构

```
PlayerSpawn (Node2D)
└── Sprite2D
    └── 显示玩家缩略图作为编辑器预览
```

- 根节点使用 **Node2D**。
- 添加 **Sprite2D** 子节点，加载玩家精灵表并设置 Hframes/Vframes 显示一帧。
- 将精灵色调设为亮薄荷绿，便于在编辑器中识别。
- 设置 `z_index = 100`，确保预览不被前景遮挡。

### 脚本要点

```gdscript
class_name PlayerSpawn
extends Node2D

@onready var player_scene := preload("uid://...") # 或拖拽生成常量引用

func _ready():
    visible = false
    await get_tree().process_frame
    var player := get_tree().get_first_node_in_group("player")
    if player:
        return
    player = player_scene.instantiate()
    get_tree().root.add_child(player)
    player.global_position = global_position
```

### 关键说明

- `await get_tree().process_frame` 等待一帧，让关卡其他脚本先初始化。
- 通过组名 `"player"` 检查是否已有玩家，避免双玩家问题。
- `visible = false` 只在游戏中隐藏，编辑器中仍可看到预览图标。
- 保存为 `general/player_spawn/player_spawn.tscn`，并附加脚本与图标。

### 使用方式

- 在每个需要测试的关卡中放置一个 `PlayerSpawn`。
- 运行时玩家会在该位置生成。
- 关卡切换后，由于玩家已存在，`PlayerSpawn` 不会重复创建。

## 3. 关卡构建建议

### 先灰盒，后美术

- 不要过早投入大量精力打磨视觉效果。
- 先用最基础的瓦片搭建可玩的关卡原型（MVP / Graybox）。
- 确认布局、跳跃距离、平台节奏后再添加背景、光照、特效。

### 围绕能力设计关卡

- 提前规划角色将获得的能力：二段跳、翻滚、冲刺、破坏地板等。
- 在关卡中预留“暂时无法到达”的区域：
  - 过高的平台 → 未来用二段跳到达。
  - 狭窄的通道 → 未来用翻滚通过。
  - 可疑的地板 → 未来用重击破坏。
- 这些“能力锁”会激励玩家回头探索。

### 逐步教学新机制

- 引入新能力时，先给玩家简单的练习场景。
- 过一两个屏幕后再增加需要组合能力（跳跃 + 冲刺）的挑战。
- 让玩家在安全的场景中理解机制，再在复杂场景中运用。

### 测试与反馈

- 自己反复跑关，检查所有平台是否可到达。
- 邀请朋友或社区成员试玩，负面反馈往往最有价值。
- 保持厚脸皮：批评某处设计不等于否定整个游戏。

## 4. 课后任务

- 修复 `LevelTransition` 的默认尺寸问题。
- 为每个关卡添加 `PlayerSpawn` 并移除关卡内的玩家节点。
- 继续扩展关卡，加入能力锁与可探索分支。
- 保持当前关卡为“垂直切片”级别的完成度，其余关卡先用灰盒搭建。
